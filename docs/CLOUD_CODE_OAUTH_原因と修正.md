Cloud Code をインストールした# Cloud Code ログイン 400 エラー：原因（可能性順）と修正手順

`harukahatanaka@colour-monster.com` で Cloud Code にサインインする際の「400. That's an error.」について、**可能性が高い順**に原因を並べ、それぞれの改善・修正手順を記載します。

---

## 画面のどこに何を入れるか（OAuth クライアント編集画面）

「ウェブアプリケーション」の OAuth 2.0 クライアントを開いた編集画面で、**どこに何を入力するか**を項目ごとに説明します。

### 1. 承認済みの JavaScript 生成元（画面の上〜中央あたり）

- **場所**: 「承認済みの JavaScript 生成元」という見出しの下にある **URI 1 \*** などのテキスト欄。
- **意味**: OAuth の認証リクエストを**送信してよいオリジン**（スキーム + ホスト + ポート）です。パスは付けません。
- **入力する値**（Cloud Code 用に以下を追加）:
  - **URI 1**: `http://localhost:3000`
  - **+ URI を追加** を押して **URI 2**: `http://127.0.0.1:3000`
  - 必要なら **URI 3**: `http://localhost`、**URI 4**: `http://127.0.0.1`
- **注意**: `https` にしない、末尾に `/` やパス（例: `/callback`）を付けない。**オリジンのみ**（例: `http://localhost:3000`）で入力。

---

### 2. 承認済みのリダイレクト URI（その下のブロック）

- **場所**: 「承認済みのリダイレクト URI」という見出しの下にある **URI 1 \*** などのテキスト欄。
- **意味**: 認証後に Google が**ブラウザを戻す先のURL**です。ここが 1 文字でも違うと 400 やリダイレクトエラーの原因になります。
- **入力する値**（Cloud Code 用に以下をすべて登録）:
  - **URI 1**: `http://localhost`
  - **+ URI を追加** → **URI 2**: `http://localhost:3000`
  - **+ URI を追加** → **URI 3**: `http://127.0.0.1`
  - **+ URI を追加** → **URI 4**: `http://127.0.0.1:3000`
- **注意**:
  - `http` / `https`、`localhost` / `127.0.0.1`、ポートの有無は**別のURI**として扱われるので、上記 4 つをすべて入れておくと安全です。
  - パス付き（例: `http://localhost:3000/callback`）を Cloud Code が使う場合は、そのURLを**そのまま**1つ追加してください。

---

### 3. クライアント シークレット（画面右側）

- **場所**: 「クライアント シークレット」という見出しの下。既にマスクされた値（例: `****SWc9`）が表示されています。
- **意味**: 認証コードをアクセストークンに交換するときに使う秘密の文字列です。
- **入力**: **何も新しく入力しません**。既存のシークレットをそのまま使います。
- **操作**: 秘密を漏らしたなどで再発行する場合だけ、「シークレットを追加」などで新しいシークレットを発行し、その値をアプリ側の設定に反映します。Cloud Code のログインだけなら通常は触りません。

---

### 4. 保存ボタン（画面左下）

- **場所**: 画面左下の青い **「保存」** ボタン。
- **操作**: 「承認済みの JavaScript 生成元」や「承認済みのリダイレクト URI」を**いじったあとは必ず「保存」を押す**。押さないと反映されません。
- **補足**: 保存後、「設定が有効になるまで 5 分から数時間かかることがあります」と出るので、変更後は少し待ってから Cloud Code で再度サインインを試してください。

---

### 入力値のクイックコピー用

**承認済みの JavaScript 生成元**

```
http://localhost:3000
http://127.0.0.1:3000
```

**承認済みのリダイレクト URI**

```
http://localhost
http://localhost:3000
http://127.0.0.1
http://127.0.0.1:3000
```

## OAuth クライアントを保存した「この後の手順」

GCP で「ウェブアプリケーションのクライアント ID」を編集し、**保存**まで完了したあとに行うことです。

### 1. 設定の反映を待つ（5〜10 分）

- 画面の注意書きのとおり、**設定が有効になるまで 5 分〜数時間**かかることがあります。
- まずは **5〜10 分ほど待ってから** 次のステップに進んでください。

### 2. Cursor で Cloud Code に再度サインインする

1. **Cursor を起動**（すでに開いている場合はそのままでよい）。
2. **コマンドパレット**を開く：`Cmd + Shift + P`（Mac）または `Ctrl + Shift + P`（Windows）。
3. **「Cloud Code: Sign in to Google Cloud」** と入力して実行。
4. ブラウザが開いたら、**harukahatanaka@colour-monster.com** でログインし、表示される権限に**許可**を選ぶ。
5. エラーが出ずに Cursor に戻り、ステータスバーなどに Cloud Code のサインイン状態が表示されれば成功です。

### 3. まだ 400 エラーが出る場合

- **リダイレクト URI を足す**  
  「承認済みのリダイレクト URI」に、次の 4 つが**すべて**入っているか確認し、足りないものだけ追加して保存してください。  
  `http://localhost` / `http://localhost:3000` / `http://127.0.0.1` / `http://127.0.0.1:3000`
- **OAuth 同意画面**  
  **API とサービス** → **OAuth 同意画面** で、**公開ステータスが「テスト」**になっているか、**テストユーザー**に **harukahatanaka@colour-monster.com** が追加されているか確認してください。
- **実際のリダイレクト URL を確認する**  
  ブラウザの開発者ツール（F12）→ **ネットワーク**タブで、エラーが出たときのリクエスト URL を開き、`redirect_uri=` の値を確認します。その URL を**そのまま**「承認済みのリダイレクト URI」に 1 つ追加して保存し、再度サインインを試してください。

### 4. サインインに成功したあと

- Cloud Code の機能（gcloud 連携、Cloud Run デプロイなど）が使えるようになります。
- ローカルで Google API を叩くアプリをデバッグする場合は、別途 **`gcloud auth application-default login`** の実行が必要な場合があります（Cloud Code のドキュメントを参照）。

## 原因の並び順（可能性が高い順）

| 順位 | 原因 |
|------|------|
| **1** | **リダイレクトURIの不一致** — Cloud Code が送るリダイレクトURIが GCP に登録されていない、または一致していない |
| **2** | **OAuth クライアントタイプの不一致** — 「デスクトップ」と「Web」の取り違えで、許可されるリダイレクトURIが異なる |
| **3** | **リダイレクトURIの形式** — `localhost` と `127.0.0.1`、ポートの有無の違い |
| **4** | **プロジェクトの公開状態** — 本番公開だと localhost が許可されない場合がある |
| **5** | **OAuth 同意画面の設定** — テストユーザー未追加など |

---

## 1. リダイレクトURIの不一致（最優先で対応）

### 何が起きているか
- Cloud Code は認証後に **`http://localhost:3000`**（または類似のポート）へリダイレクトします（公式ドキュメントで言及あり）。
- 現在の `credentials.json` は **「デスクトップアプリケーション」**（`installed`）タイプで、`redirect_uris` は **`http://localhost`** のみです。
- **重要：** Cloud Code は **「Webアプリケーション」** タイプのOAuthクライアントを使用するため、ポート番号付きのリダイレクトURI（例: `http://localhost:3000`）が必要です。
- デスクトップアプリケーションタイプでは、Google Cloud Console でカスタムリダイレクトURI（ポート番号付き）を設定できません。
- 実際に使われる URI が Google Cloud Console に登録されていないと、**400 Bad Request（malformed）** になりやすいです。

### 修正手順（Google Cloud Console で実施）

**重要：** この手順は **「Webアプリケーション」** タイプのOAuthクライアントに対して実施してください。

1. [Google Cloud Console](https://console.cloud.google.com/) にログインし、プロジェクト **western-storm-486708-m5** を選択。
2. **API とサービス** → **認証情報** を開く。
3. **「ウェブアプリケーション」** タイプの OAuth 2.0 クライアント ID を探す。
   - もし存在しない場合は、**原因2の手順** に従って「ウェブアプリケーション」タイプのクライアントを新規作成してください。
4. 該当するクライアントをクリックして編集画面を開く。
5. **承認済みのリダイレクト URI** に、以下を**すべて**追加（既にあるものはそのまま）。
   - `http://localhost`
   - `http://localhost:3000`
   - `http://127.0.0.1`
   - `http://127.0.0.1:3000`
6. **保存** する。
7. 反映に 5 分〜数時間かかることがあるため、しばらく待ってから再度 Cloud Code でサインインを試す。

**注意：** 現在の `credentials.json` のクライアント（デスクトップアプリケーション）は、このプロジェクトの Google Drive 連携等で使用しているため、**そのままにしておいてください**。Cloud Code 用には別途「ウェブアプリケーション」タイプのクライアントが必要です。

※ Cloud Code が別のポートを使う場合は、ブラウザの開発者ツール（ネットワークタブ）でエラー時のリクエスト URL を確認し、表示されている `redirect_uri` をそのまま「承認済みのリダイレクト URI」に追加してください。

## 2. OAuth クライアントタイプの不一致

### 何が起きているか
- 現在の `credentials.json` は **「デスクトップアプリ」**（`installed`）用です。
- Cloud Code が **「Web アプリケーション」** としてリクエストを送っている場合、リダイレクトURIの扱いや検証が変わり、400 の原因になります。

### 修正手順（Google Cloud Console で実施）

1. **API とサービス** → **認証情報** を開く。
2. **+ 認証情報を作成** → **OAuth クライアント ID** を選択。
3. **アプリケーションの種類** で **「ウェブアプリケーション」** を選択。
4. **名前** を任意で入力（例: Cloud Code Web）。
5. **承認済みのリダイレクト URI** に、以下を追加。
   - `http://localhost:3000`
   - `http://127.0.0.1:3000`
   - （必要に応じて）`http://localhost` / `http://127.0.0.1`
6. **作成** して保存。

既存の「デスクトップ」用クライアントは、このプロジェクトの他の用途（例: 本プロジェクトの `credentials.json`）で使う場合はそのままにし、**Cloud Code 用に「ウェブアプリケーション」を 1 つ追加**する形にすると安全です。

---

## 3. リダイレクトURIの形式

### 何が起きているか
- `http://localhost` と `http://127.0.0.1` は Google では**別の URI** として扱われます。
- ポート付き（例: `:3000`）とポートなしでも別扱いです。
- 実際に送られている形式が Console に無いと 400 になります。

### 修正手順
- 上記 **1** の手順で、`localhost` / `127.0.0.1` の両方と、ポートあり・なしの両方を登録しておけば、形式の違いによる不整合は防げます。
- 既に 1 を実施していれば、この項目は同じ設定で解消されています。

---

## 4. プロジェクトの公開状態

### 何が起きているか
- OAuth 同意画面が **「本番」** に設定されていると、一部のシナリオで localhost が許可されにくくなることがあります。
- **「テスト」** の場合は、テストユーザーに追加されたアカウントで localhost を使ったログインが可能です。

### 修正手順（Google Cloud Console で実施）

1. **API とサービス** → **OAuth 同意画面** を開く。
2. **公開ステータス** が **「テスト」** になっているか確認。
3. 本番になっている場合で、開発・検証のみ行う場合は **「テスト」** に変更して保存。

本番のまま運用する必要がある場合は、リダイレクトURIの登録（1・2）とクライアントタイプ（2）の見直しを優先してください。

---

## 5. OAuth 同意画面の設定

### 何が起きているか
- **テスト** モードでは、**テストユーザー** に追加されていないアカウントはログインできません。
- `harukahatanaka@colour-monster.com` がテストユーザーに含まれていないと、同意画面でブロックされる可能性があります（400 とは別のエラーになることもあります）。

### 修正手順（Google Cloud Console で実施）

1. **API とサービス** → **OAuth 同意画面** を開く。
2. **テストユーザー** セクションで **+ ADD USERS** をクリック。
3. **harukahatanaka@colour-monster.com** を追加して保存。

これで、テストモードのまま当該アカウントで Cloud Code にサインインできるようになります。

---

## 実施順序の目安

**重要：** Cloud Code は「ウェブアプリケーション」タイプのOAuthクライアントを使用します。現在の `credentials.json` は「デスクトップアプリケーション」タイプのため、**別途「ウェブアプリケーション」タイプのクライアントが必要**です。

1. **まず 2（ウェブアプリケーション用クライアントの作成）** を実施して、新しいクライアントを作成する。
2. 次に **1（リダイレクトURIの追加）** を実施し、作成したウェブアプリケーション用クライアントにリダイレクトURIを追加する。
3. 数分待ってから再度 Cloud Code でサインインを試す。
4. まだ 400 が出る場合は **4（テストモード）** と **5（テストユーザー追加）** を実施。

プロジェクトの `credentials.json` は、このプロジェクト内の Google Drive 連携等で使うためのものです。Cloud Code のサインインは、**同じ GCP プロジェクトの「認証情報」と「OAuth 同意画面」の設定**で決まるため、上記はすべて **Google Cloud Console 上での変更** で対応します。
